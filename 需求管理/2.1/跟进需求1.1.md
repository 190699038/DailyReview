# 跟进事项汇报 — 产品需求文档 v1.1

> 版本：v1.1  
> 日期：2026-03-15  
> 需求来源：CEO每日汇报模板  
> 变更说明：优化为三大类结构，子任务支持按地区拆分，去掉跨部门协调事项

---

## 一、需求概述

基于 CEO 每日汇报模板，开发「跟进事项汇报」功能模块。三大板块结构：

| 板块 | 说明 |
|------|------|
| **今日业务事项进展** | 支付平台、S级需求（按地区拆分）、投放攻坚、产品核心指标等 |
| **今日AI事项进展** | AI全员使用、AI工具库等 |
| **风险与异常预警** | 🔴 紧急 / 🟡 关注，支持动态增减 |

**v1.1 核心变更：**
1. 去掉「跨部门协调事项」板块
2. 每个大类下的子任务支持**动态增减**
3. S级需求等子任务支持**按地区拆分**为多条独立记录（美国、欧洲、澳洲等各有独立进展）,每天打开页面后自动同步(select * from weekly_goals where mondayDate = '20260316' and priority = 10 and department_id  in (2,3)) -- mondayDate 本周周一的时间 , 地区为country字段  
4. 支持手动创建S级的任务，查询昨日任务时，支持单任务复制到今日

## 二、数据库设计（MySQL）

### 2.1 主表：`follow_up_reports`（跟进汇报主表）

> **v1.1 变更：** 移除 `coordination_matters` 字段

```sql
CREATE TABLE `follow_up_reports` (
  `id` int(11) NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `reporter` varchar(50) NOT NULL COMMENT '汇报人',
  `report_date` int(8) NOT NULL COMMENT '汇报日期 YYYYMMDD',
  `week_day` varchar(10) DEFAULT NULL COMMENT '星期几（周一~周日）',
  `status` tinyint(1) NOT NULL DEFAULT 0 COMMENT '状态 0=草稿 1=已提交',
  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_reporter_date` (`reporter`, `report_date`),
  KEY `idx_report_date` (`report_date`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='跟进事项汇报主表';
```

### 2.2 明细表：`follow_up_report_items`（汇报事项明细）

> **v1.1 变更：** 新增 `sub_title` 字段，用于地区拆分

```sql
CREATE TABLE `follow_up_report_items` (
  `id` int(11) NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `report_id` int(11) NOT NULL COMMENT '关联主表ID',
  `category` varchar(30) NOT NULL COMMENT '分类：business=业务事项, ai=AI事项, risk=风险预警',
  `section_title` varchar(100) NOT NULL COMMENT '模块标题（如：支付平台、S级需求、AI全员使用）',
  `sub_title` varchar(100) DEFAULT NULL COMMENT '子标题/地区（如：美国、欧洲、澳洲），无地区拆分时为NULL',
  `responsible_person` varchar(100) DEFAULT NULL COMMENT '负责人',
  `section_order` int(3) NOT NULL DEFAULT 0 COMMENT '排序序号',
  `content` text NOT NULL COMMENT '事项内容（JSON格式存储多个字段）',
  `risk_level` tinyint(1) NOT NULL DEFAULT 0 COMMENT '风险等级 0=无 1=黄色关注 2=红色紧急',
  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  KEY `idx_report_id` (`report_id`),
  KEY `idx_category` (`category`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='汇报事项明细表';
```

### 2.3 v1.1 数据库变更 SQL（增量）

```sql
-- 主表：移除跨部门协调字段
ALTER TABLE `follow_up_reports` DROP COLUMN `coordination_matters`;

-- 明细表：新增 sub_title 字段
ALTER TABLE `follow_up_report_items` ADD COLUMN `sub_title` varchar(100) DEFAULT NULL COMMENT '子标题/地区' AFTER `section_title`;
```

### 2.4 数据结构说明

#### 三级数据层次

```
follow_up_reports（主表 - 一天一条）
  └── follow_up_report_items（明细 - 每个子任务一条）
        ├── category = "business"
        │     ├── section_title = "支付平台", sub_title = NULL       → 1条记录
        │     ├── section_title = "S级需求",  sub_title = "美国"     → 1条记录
        │     ├── section_title = "S级需求",  sub_title = "欧洲"     → 1条记录
        │     ├── section_title = "S级需求",  sub_title = "澳洲"     → 1条记录
        │     ├── section_title = "投放攻坚", sub_title = NULL       → 1条记录
        │     └── section_title = "产品核心指标", sub_title = NULL   → 1条记录
        ├── category = "ai"
        │     ├── section_title = "AI全员使用", sub_title = NULL     → 1条记录
        │     └── section_title = "AI工具库",   sub_title = NULL     → 1条记录
        └── category = "risk"
              ├── risk_level = 2, content = {"description":"..."}    → 紧急
              └── risk_level = 1, content = {"description":"..."}    → 关注
```

#### content JSON 格式（按 section_title 区分）

| section_title | sub_title | content 示例 |
|---------------|-----------|-------------|
| 支付平台 | NULL | `{"进展":"已对接","风险/卡点":"无"}` |
| S级需求 | 美国 | `{"开发进度":"已完成80%","测试进度":"未开始","是否有B级任务混入":"否"}` |
| S级需求 | 欧洲 | `{"开发进度":"进行中","测试进度":"未开始","是否有B级任务混入":"否"}` |
| 投放攻坚 | NULL | `{"攻坚进展":"解决回调延迟","投放数据":"消耗12W"}` |
| 产品核心指标 | NULL | `{"裂变":"15%","新增付费率":"8.2%","一付到二付":"32%"}` |
| AI全员使用 | NULL | `{"管理层方案收集":"已提交3人","游戏技术组AI演练":"第3天"}` |
| AI工具库 | NULL | `{"新增工具":"报表工具","使用问题汇总":"收集3个解决2个"}` |
| （风险项） | NULL | `{"description":"服务器过载"}` |

## 三、子任务模板配置

### 3.1 模板定义

前端预定义子任务模板，控制每个子任务的填写字段。模板可在前端静态配置，后续可扩展为后端动态配置。

#### 业务事项模板（category = business）

| section_title | 是否支持地区拆分 | 填写字段 | 字段类型 |
|---------------|:---------------:|----------|---------|
| 支付平台 | ❌ | 进展、风险/卡点 | textarea × 2 |
| S级需求 | ✅ | 开发进度、测试进度、是否有B级任务混入 | textarea × 2 + select |
| 投放攻坚 | ❌ | 攻坚进展、投放数据 | textarea × 2 |
| 产品核心指标 | ❌ | 裂变、新增付费率、一付到二付 | textarea × 3 |

> 💡 支持地区拆分的子任务：用户可从 `project_groups`（category='Region'）中选择地区，每个地区生成一条独立的明细记录，各自填写独立的进展内容。

#### AI事项模板（category = ai）

| section_title | 是否支持地区拆分 | 填写字段 | 字段类型 |
|---------------|:---------------:|----------|---------|
| AI全员使用 | ❌ | 管理层方案收集、游戏技术组AI演练 | textarea × 2 |
| AI工具库 | ❌ | 新增工具、使用问题汇总 | textarea × 2 |

#### 风险预警（category = risk）

| 项目 | 说明 |
|------|------|
| 风险项 | 动态增减行，每行包含：事项描述(textarea) + 风险等级(select: 🔴紧急/🟡关注) |

### 3.2 子任务动态增减

三个大类下的子任务均支持**动态新增和删除**：

- **业务事项**：默认显示 4 个预定义子任务（支付平台/S级需求/投放攻坚/产品核心指标），用户可以：
  - 新增自定义子任务（输入标题、选择是否按地区拆分、定义填写字段）
  - 删除不需要的子任务
  
- **AI事项**：默认显示 2 个预定义子任务，可动态增减

- **风险预警**：每条风险项独立增减

## 四、后端 API 设计

### 4.1 接口文件：`FollowUpReportAPI.php`

| 操作 | action 值 | 请求方式 | 参数 | 说明 |
|------|----------|---------|------|------|
| 获取汇报列表 | `list` | GET | `report_date`, `reporter`（可选） | 按日期查询汇报列表 |
| 获取汇报详情 | `get` | GET | `id` 或 `report_date` + `reporter` | 获取含明细的完整汇报 |
| 创建/保存汇报 | `save` | POST | 主表+明细数据（JSON body） | 新增或更新（upsert） |
| 提交汇报 | `submit` | POST | `id` | 将草稿改为已提交 |
| 删除汇报 | `delete` | POST | `id` | 删除汇报及关联明细 |
| 复制历史汇报 | `copy` | POST | `source_date`, `target_date`, `reporter` | 复制某天模板到新日期 |
| 导出 Markdown | `export_md` | GET | `id` 或 `report_date` + `reporter` | 导出为 Markdown 文本 |
| 获取地区列表 | `get_regions` | GET | 无 | 从 project_groups 获取 Region 类型列表 |
