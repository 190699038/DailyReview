# 跟进事项汇报 — 产品需求文档 v2.1.1

> 版本：v2.1.1  
> 日期：2026-03-16  
> 需求来源：汇报格式.md（CEO 每日跟进汇报简化模板）  
> 变更说明：大幅简化页面结构，仅保留 **S级任务总览** 和 **跟进事项** 两大板块，去除业务子模板、AI事项、风险预警板块

---

## 一、需求概述

基于简化版 CEO 每日汇报格式，将现有「跟进事项汇报」模块从 4 大板块精简为 **2 大板块**：

| 板块 | 说明 |
|------|------|
| **业务部门S级任务** | 从 `weekly_goals` 表自动同步，按部门分组展示，含自动统计汇总 |
| **跟进事项** | 手动管理的跟进事项列表，每项含 状态、今日进度、下一步、异常汇报 |

**v2.1.1 核心变更（相对 v1.1）：**
1. **去掉** 业务事项板块（支付平台、投放攻坚、产品核心指标等子模板配置）
2. **去掉** AI事项板块
3. **去掉** 风险与异常预警板块
4. **保留并优化** S级任务：改为按**部门分组**展示（游戏技术组 / 产品组 / 奇胜技术组），从 `weekly_goals` 自动同步，只读展示 + 自动统计
5. **保留并优化** 跟进事项：每项支持 **状态、今日进度、下一步、异常汇报** 四个输入字段，支持增删改

## 二、数据库设计（MySQL）

### 2.1 沿用表：`follow_up_reports`（主表 — 无变更）

```sql
CREATE TABLE `follow_up_reports` (
  `id` int(11) NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `reporter` varchar(50) NOT NULL COMMENT '汇报人',
  `report_date` int(8) NOT NULL COMMENT '汇报日期 YYYYMMDD',
  `week_day` varchar(10) DEFAULT NULL COMMENT '星期几',
  `status` tinyint(1) NOT NULL DEFAULT 0 COMMENT '0=草稿 1=已提交',
  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_reporter_date` (`reporter`, `report_date`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='跟进事项汇报主表';
```

### 2.2 沿用表：`follow_up_report_items`（明细表 — 字段精简）

> **v2.1.1 变更：**

> - `sub_title`、`responsible_person`、`risk_level` 字段保留但不再使用，兼容旧数据  
> - `content` JSON 格式统一为 `{status, progress, next_step, exception}`

## 三、数据结构说明

### 3.1 简化后的数据层次

```
follow_up_reports（主表 - 一天一条）
  └── follow_up_report_items（明细 - 仅存跟进事项）
        └── category = "followup"
              ├── section_title = "AI攻坚(工具)碰头"
              ├── section_title = "谷歌包上架流程跟进"
              ├── section_title = "支付平台进展（SF平台）"
              └── ...

weekly_goals（S级任务 - 实时查询，不存入明细表）
  ├── department_id = 2  → 游戏技术组
  ├── department_id = 3  → 产品组
  └── department_id = 16 → 奇胜技术组
```

### 3.2 S级任务展示格式

按部门分组，每组头部自动统计：
```
本周 游戏技术组 已完成S级任务 2个，未完成 11个，今日完成 2个，暂停 2个
```

每条S级任务的展示字段（来自 weekly_goals）：
```
序号、【S】任务名 - 地区名 - 进度% - 部门名 - 执行人 - 期限标签 - 状态
```

示例：
```
1、【S】 美国三H5 - 支付刷单风控 - 美国3 - 0% - 游戏技术组 - 梁嘉轩/陶俊华 - 当周完成 - 进行中
```

## 四、后端 API 设计

### 4.1 接口文件：`FollowUpReportAPI.php`

> **v2.1.1 变更：** 精简 API，移除不再需要的 `get_regions` 接口，优化 `sync_s_goals` 按部门分组

| 操作 | action 值 | 请求方式 | 参数 | 说明 |
|------|----------|---------|------|------|
| 获取汇报详情 | `get` | GET | `report_date` + `reporter` | 获取含跟进事项明细的完整汇报 |
| 保存汇报 | `save` | POST | JSON body | 保存跟进事项（upsert） |
| 删除汇报 | `delete` | POST | `id` | 删除汇报及关联明细 |
| 复制昨日 | `copy` | POST | `source_date`, `target_date`, `reporter` | 复制昨日跟进事项到今日 |
| 获取S级任务 | `sync_s_goals` | GET | `monday_date` | 按部门分组获取本周S级任务 + 统计数据 |
| 获取部门列表 | `get_departments` | GET | 无 | 获取 department_id → 部门名称映射 |
| 导出Markdown | `export_md` | GET | `report_date` + `reporter` | 导出完整汇报为 Markdown |

### 4.2 接口详情

#### 4.2.1 获取S级任务 `GET ?action=sync_s_goals&monday_date=20260316`

**查询逻辑：**
```sql
SELECT wg.id, wg.weekly_goal, wg.executor, wg.country, wg.department_id,
       wg.status, wg.process
FROM weekly_goals wg
WHERE wg.mondayDate = ? AND wg.priority = 10
  AND wg.department_id IN (2, 3, 16)
ORDER BY wg.department_id, wg.id
```

### 5.3 交互逻辑

| 场景 | 逻辑 |

| 进入页面 | 1. 调用 `sync_s_goals` 获取S级任务展示 2. 调用 `get` 加载当天跟进事项 |

| 导出MD | 合并 S级任务（实时查询）+ 跟进事项（已保存数据）输出完整 Markdown |
