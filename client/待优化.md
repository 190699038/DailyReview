# 周目标管理系统 — 待优化项清单

> 生成日期：2026-03-12  
> 基于前端、后端、数据库三端源码分析  
> 最后更新：2026-03-12（已完成首批优化）

---

## 一、安全类（高优先级 🔴）

### ✅ 1.1 SQL 注入漏洞（已修复）

**文件：** `server/DayTaskAPI.php`  
**修复：** `getUserTodayTask` 已改为 PDO 参数化查询，`uids` 使用 `intval` 过滤后生成占位符，`start_date`/`end_date` 使用 `intval` 转换。

---

### ✅ 1.2 敏感信息硬编码（已修复）

**修复：** 创建了 `server/.env` 环境变量文件和 `server/env_loader.php` 加载器。所有敏感信息（数据库凭据、钉钉 Webhook Token、OA 账号密码）已从代码中移除，改为通过 `getenv()` 读取。已添加 `.gitignore` 防止 `.env` 文件被提交。

---

### 1.3 缺少身份认证与权限控制

**问题：** 所有 API 接口完全开放，无需登录即可访问。任何人获取到 API 地址后可以：
- 读取所有部门的目标和任务数据
- 创建/修改/删除任意目标
- 触发钉钉消息推送

**建议：**
- 增加 Token/Session 身份认证中间件
- 按部门和角色限制数据访问范围
- 敏感操作（删除、推送）增加权限校验

---

### ✅ 1.4 CORS 过于宽松（已修复）

**修复：** 所有 PHP 文件的 CORS 头已从 `*` 改为从 `.env` 读取 `CORS_ORIGIN`，默认限制为 `https://daily.gameyzy.com`。

---

### ✅ 1.5 SSL 验证关闭（已修复）

**修复：** `OAIntegrationAPI.php` 中 `CURLOPT_SSL_VERIFYPEER` 已从 `false` 改为 `true`。

---

### ✅ 1.6 WeekGoalAPI SQL 注入漏洞（已修复）

**文件：** `server/WeekGoalAPI.php`  
**修复：** `list` action 中 `mondayDates` 参数原来直接拼入 SQL `IN()` 子句，已改为 `intval` 过滤 + 占位符参数化查询。

---

### ✅ 1.7 前端 OA 凭证硬编码（已修复）

**文件：** `client/src/utils/dailyPlanAsync.js`  
**修复：** 移除硬编码的 OA 账号密码。OA 登录已在后端 `OAIntegrationAPI.php` 通过 `.env` 处理。

---

### ✅ 1.8 DingDingSend 输入注入与信息泄露（已修复）

**文件：** `server/DingDingSend.php`  
**修复：**
- 添加必填字段校验（priority, country, type, uid, describe, customer, time）
- 所有用户输入使用 `htmlspecialchars` 过滤，防止注入
- 移除返回中的 `dingtalk_response` 原始响应，避免信息泄露
- 移除生产环境调试日志（请求头、原始输入写文件）

## 二、架构类（中高优先级 🟠）

### 2.1 daily_tasks 与 daily_tasks_today 表结构完全重复

**问题：** 两张表字段完全一致，仅区分"昨日同步"和"今日同步"场景。数据冗余，查询逻辑复杂。  
**建议：** 合并为一张表，增加 `sync_type` 字段（yesterday/today）或 `sync_date` 字段区分。

---

### 2.2 executor_id 存储风格不统一

| 表 | executor_id 类型 | 存储方式 |
|----|-----------------|---------|
| weekly_goals | TEXT | 多人用 `/` 分隔 |
| daily_goals | INT | 单人单行 |
| daily_tasks | INT | 单人单行 |
| users | INT (PK) | 单值 |

**问题：** weekly_goals 中用 TEXT 存储多个 ID 违反第一范式，导致查询不便、关联困难。  
**建议：** 引入 `weekly_goal_executors` 关联表，实现多对多关系。

---

### 2.3 缺少统一状态管理（前端）

**问题：** 前端使用 localStorage + 组件级 ref/reactive 管理状态，缺乏全局状态管理，导致：
- 部门切换需要在多个组件间手动传递
- 用户数据重复加载
- 缓存一致性难以维护

**建议：** 引入 Pinia 状态管理，统一管理用户信息、部门信息、主题配置等全局状态。

---

### 2.4 后端无框架，接口风格不统一

**问题：**
- 每个 PHP 文件独立处理路由（`$_REQUEST['action']`）
- GET/POST 混用（`$_REQUEST` 同时接受 GET 和 POST）
- 返回格式不统一（有些返回 `{data: []}`, 有些返回 `{id: 1}`, 有些返回裸数组）
- 无统一的错误响应格式

**建议：**
- 统一 RESTful 接口规范：`GET /api/weekly-goals`, `POST /api/weekly-goals`, `PUT /api/weekly-goals/:id`
- 统一返回格式：`{ code: 0, data: {}, message: "success" }`
- 引入轻量 PHP 路由（如 Slim Framework）或统一中间件处理

---

### 2.5 缺少请求速率限制

**问题：** API 无请求频率限制，存在被刷接口和恶意调用风险。  
**建议：** 增加 IP/Token 级别的 Rate Limiting。

---

## 三、前端优化（中优先级 🟡）

### 3.1 WeekTask.vue 组件过大（1000+ 行）

**问题：** 单文件包含筛选逻辑、表格渲染、弹窗表单、状态操作、剪贴板、钉钉推送等大量逻辑，可读性和可维护性差。  
**建议：**
- 拆分为 `WeekTaskFilter.vue`（筛选区）、`WeekTaskTable.vue`（表格）、`WeekTaskForm.vue`（新增/编辑弹窗）
- 将业务逻辑抽取为 `useWeekTask.js` 组合式函数

---

### 3.2 批量导入组件冗余

**问题：** `WeekTaskBatch.vue`、`WeekTaskBatchGame.vue`、`WeekTaskBatchProduct.vue` 三个组件逻辑高度相似，仅列映射规则不同。  
**建议：** 合并为一个通用的 `WeekTaskBatchImport.vue`，通过配置对象传入列映射规则：
```js
const columnMaps = {
  game: { priority: 'A', content: 'C', executor: 'D' },
  product: { project: 'B', content: 'C', executor: 'H' },
  qisheng: { priority: 'A', project: 'B', content: 'C', executor: 'D' }
}
```

---

### ✅ 3.3 DayTaskInfo 存在冗余文件（已清理）

**修复：** 已删除 3 个未使用组件：
- `components/DayTaskInfo copy.vue`（拷贝文件）
- `components/DayTaskInfo.vue`（未被任何组件引用）
- `components/HelloWorld.vue`（Vue 示例模板）

---

### ✅ 3.4 API 调用缺少统一错误处理（已修复）

**修复：** `http.js` 响应拦截器已增强：
- 成功响应中检测 `data.error` 字段并自动弹出 ElMessage 错误提示
- 401 状态码跳转登录
- 500 状态码显示服务器错误
- 统一处理网络错误和超时

---

### ✅ 3.5 localStorage JSON.parse 空值崩溃（已修复）

**修复：** `WeekGoal.vue`、`DailyHistory.vue`、`WeekTask.vue` 中 `JSON.parse(localStorage.getItem(...))` 缺少空值检查，当缓存不存在时导致运行时错误。已统一加入 `cache ? JSON.parse(cache) : []` 保护。

---

### ✅ 3.6 TestTaskStatics 内存泄漏（已修复）

**文件：** `client/src/view/TestTaskStatics.vue`  
**修复：** 匿名 `resize` 事件监听器在组件卸载后不会被移除，导致内存泄漏。已改为命名函数并在 `onBeforeUnmount` 中清理。

---

### ✅ 3.7 未使用的 import 清理（已修复）

**修复：**
- `excelParser.js` 移除未使用的 `id` 导入
- `dateUtils.js` 移除未使用的 `defineProps, defineEmits`
- `useTheme.js` 移除未使用的 `watch`

---

### ✅ 3.8 路由路径不一致（已修复）

**文件：** `client/src/router/index.js`  
**修复：** `user-today-plan` 路由使用了绝对路径 `/user-today-plan`（带前导 `/`），与其他子路由不一致。已改为相对路径 `user-today-plan`。

---

### ✅ 3.9 console.log 调试语句清理（已修复）

**修复：** 清理了 32 处调试用的 `console.log` 语句（跨 12 个前端文件）。保留了 `catch` 块中的 `console.error` 错误日志。

---

### 3.10 日期格式处理分散

**问题：** 多处手动处理日期格式（YYYYMMDD 整数），存在以下问题：
- `dateUtils.js` 与各组件中存在重复的日期处理逻辑
- Excel 导入时的日期解析逻辑分散在多个组件中

**建议：** 统一封装日期工具，支持 YYYYMMDD 整数与 Date/dayjs 对象的双向转换。可引入 dayjs 简化处理。

---

### 3.11 无加载状态与空状态提示

**问题：** 部分页面在数据加载中和无数据时缺少友好提示。  
**建议：** 增加 `v-loading` 和 `el-empty` 组件处理加载态与空态。

---

### 3.12 前端缺少表单验证

**问题：** 部分表单提交时缺少前端校验（如必填项、日期格式等），依赖后端校验。  
**建议：** 使用 Element Plus 的 `el-form` rules 规则配置前端验证。

## 四、后端优化（中优先级 🟡）

### ✅ 4.1 循环查询性能问题（已修复）

**修复：** `DayTaskAPI.php` 中 `getUserGoalAndTasks` 和 `get_history` 两个 action 已从逐日/逐人循环查询改为 `WHERE IN (...)` 批量查询，然后在 PHP 端按日期/执行人分组。同时修复了重复的 `delete` case。

原来一周 7 天执行 21 次 SQL → 现在仅执行 3 次 SQL。

---

### 4.2 OA 同步逐用户串行查询

**文件：** `server/OAIntegrationAPI.php`  
**问题：** 逐用户调用 OA 接口获取日计划，用户多时耗时严重。  
**建议：**
- 使用 `curl_multi` 并行请求
- 增加同步状态记录，支持增量同步
- 添加失败重试机制

---

### ✅ 4.3 OA 调试输出残留（已修复）

**文件：** `server/OAIntegrationAPI.php`  
**修复：** 移除了 `syncDaily` 函数中两处 `echo` 调试输出（直接输出 `oa_task_id` 和 `executor_id`），避免干扰正常 JSON 响应。

---

### 4.4 缺少操作日志/审计

**问题：** 创建、修改、删除操作无日志记录，出问题无法追溯。  
**建议：** 增加 `operation_logs` 表，记录操作人、操作类型、操作时间、操作内容。

---

### 4.5 WeekTaskSend.php 硬编码业务规则

**问题：**
- 特定人员过滤：`executor NOT LIKE '%梁超%'`
- 特定 UserID 硬编码
- 部门 emoji 映射硬编码

**建议：** 将业务规则配置化，存入数据库或配置文件，便于维护。

---

### 4.6 ProductReview 使用文件存储而非数据库

**文件：** `server/ProductReview.php`  
**问题：** 产品需求数据存储为 JSON 文件（`server/product/{日期}.json`），而非数据库，难以查询和管理。  
**建议：** 建立 `product_reviews` 表统一存储。

---

### 4.7 POST 数据读取方式不一致

| 文件 | 读取方式 |
|------|---------|
| WeekGoalAPI | `$_POST` |
| DayTaskAPI | `$_REQUEST`（GET/POST混合） |
| UserInfoAPI | `$_REQUEST` + `bindParam` |
| DayGoalAPI | `json_decode(file_get_contents('php://input'))` |

**建议：** 统一为 `json_decode(file_get_contents('php://input'))` 读取 JSON 请求体，与前端 Axios 的 POST 格式一致。

## 五、数据库优化（中优先级 🟡）

### 5.1 日期字段类型不合理

**问题：** `mondayDate`、`createdate`、`date`、`pre_finish_date`、`real_finish_date` 均使用 INT 类型存储 YYYYMMDD 格式。导致：
- 无法使用数据库原生日期函数（DATE_ADD, DATEDIFF 等）
- 日期范围查询不够直观
- 无法利用日期类型索引优化

**建议：** 改为 DATE 类型，应用层做格式转换。或至少统一格式且在注释中说明。

---

### 5.2 缺少外键约束

**问题：** 表间关系完全依赖应用层维护，无数据库外键约束：
- `daily_goals.weekly_goals_id` → `weekly_goals.id`（无外键）
- `daily_tasks.executor_id` → `users.id`（无外键）
- `test_tasks_info.task_id` → `test_tasks.id`（无外键）
- `watch_user.executor_id` → `users.id`（无外键）

**风险：** 可能出现孤儿数据，删除周目标后日目标残留。  
**建议：** 增加外键约束并设置 `ON DELETE CASCADE` 或 `ON DELETE SET NULL`。

---

### 5.3 weekly_goals 缺少 department_id 单独索引

**问题：** 现有索引 `(mondayDate, real_finish_date, department_id)` 无法高效支持仅按 `department_id` 查询的场景。  
**建议：** 增加 `idx_department_id` 单独索引。

---

### 5.4 today_target 表 UNIQUE 约束不完整

**问题：** `report_date` 上有 UNIQUE 约束，但表中有 `department_id` 字段，意味着同一天只能有一个目标汇总，不支持多部门。  
**建议：** 将 UNIQUE 约束改为 `(report_date, department_id)` 联合唯一。

---

### 5.5 test_tasks_info 时间字段使用 VARCHAR

**问题：** `pre_submission_time`、`submission_time`、`planned_online_time`、`actual_online_time` 均为 VARCHAR(20)，不利于时间范围查询和排序。  
**建议：** 改为 DATE 或 DATETIME 类型。

---

### 5.6 缺少软删除机制

**问题：** 删除操作为物理删除（`DELETE FROM`），数据不可恢复。  
**建议：** 增加 `deleted_at` 字段实现软删除，保留历史数据。

---

### 5.7 缺少 created_at / updated_at 时间戳

**问题：** 大部分表缺少标准的创建时间和更新时间戳字段。  
**建议：** 增加 `created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP` 和 `updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`。

---

## 六、用户体验优化（低优先级 🟢）

### 6.1 导航中隐藏页面的入口不直观

**问题：** `daily-goal`、`history-daily`、`user-today-plan` 三个路由已注册但导航栏隐藏，用户无法自然发现。  
**建议：** 评估这些页面是否仍需要，若需要则加入导航或提供明确入口。

---

### 6.2 无数据导出功能

**问题：** 系统支持 Excel 导入但不支持导出，用户需要手动复制数据。  
**建议：** 增加周目标、日任务、测试任务的 Excel 导出功能。

---

### 6.3 无实时通知机制

**问题：** 状态变更、任务分配等无实时通知，依赖手动刷新。  
**建议：** 可考虑 WebSocket 或轮询通知关键事件。

---

### 6.4 统计图表缺少数据钻取

**问题：** 统计图表点击后显示明细列表，但无法进一步钻取（如按时间段对比）。  
**建议：** 增加多维度交叉分析和时间对比功能。

---

### 6.5 移动端体验优化空间

**问题：** 虽然支持响应式，但大量表格在手机端体验一般。  
**建议：** 移动端使用卡片列表代替表格展示，优化操作交互。
